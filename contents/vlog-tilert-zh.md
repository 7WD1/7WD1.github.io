**TileRT：把GPU算力从execution boundary里捞出来**

做推理引擎这几年，我反复看到一个现象：硬件规格书上写着理论带宽 38 TB/s，你的模型激活参数 42GB，手算一下理论上限 1000 tok/s，结果跑起来一看——几十。差了一个数量级，不是 10% 的优化空间，是整整 10 倍。

这不是新问题。2023 年大家关心能不能把模型塞进 GPU，2024 年关心连续批处理能把吞吐做到多高，到了 2025 年 agentic 时代一来，用户等的是下一个 token、下一个工具调用、下一个推理步骤。没人关心你一小时能吐多少 token 了，他们关心的是「我按下回车之后，第一个字什么时候出来」。

TileRT 团队把这个讲得很透：延迟正在成为新的智能。不是修辞——Test-Time Scaling 下，速度直接决定模型能在固定延迟预算里走多深的推理路径。推理速度不再只是成本指标，它本身就是能力的一部分。

问题的根在哪？TileRT 的 blog 里有一句话让我反复读了好几遍：GPU 不缺算力。算力被困在执行边界之间。

传统推理框架的执行模型是 `graph → operator → kernel`。每个算子单独启动、同步、做一轮内存往返。这个抽象在训练时代完全没问题——kernel 足够大，启动开销、同步开销、调度开销全部被计算摊掉了。但到了 decode，batch size 压到 1，attention kernel 的生命周期掉进微秒量级。启动间隙、跨 kernel 屏障、数据 spill、TP 同步——这些以前被批量隐藏的开销，全回来了。

Profiler 里你看到的是一个诡异的现象：kernel 还没完全热身就已经结束了。GPU 在反复执行 `launch → load → compute → store → sync`，每个边界都在打断数据流，破坏局部性，强制重新同步。瓶颈不在 GEMM 跑得多快，而在下一个阶段能不能立刻开始。

TileRT 的解法很激进，但我认为方向是对的：别再反复启动 kernel 了。把整个模型在编译期静态展开成一个 persistent Engine Kernel，Host 只启动一次，执行一直留在 GPU 上。

这个思路的执行细节值得拆开看。不是简单地把所有 kernel 串成一个大的——那样只是把问题换个形式。TileRT 把算子拆解成 tile-level task，调度到不同的 warp group 上。不同的 warp group 承担不同职责：异步数据搬运、张量计算、通信重叠。以前是 `load → barrier → compute → barrier`，现在是这些阶段在 tile 粒度上持续重叠。中间结果留在寄存器、共享内存和 L2 cache 里，不再反复写回全局内存。

从 profiler 的视角看，差别是根本性的：GPU 不再像一个反复启动 kernel 的设备，更像一条持续运行的执行流水线。

但这只解决了单卡内的问题。到 8×NVL 拓扑下，另一个限制浮现：同构并行本身。

大部分张量并行框架假设所有 GPU rank 执行相同的逻辑。训练时代这很自然——计算模式规则，同步行为稳定。推理打破了这个假设。稀疏路由、Top-K 选择、动态索引、长上下文 attention、MTP——越来越多执行阶段不再适合均匀分配到每一张卡上。强制所有 rank 执行相同逻辑，带来的是冗余计算、过度广播和同步放大。

TileRT 的做法是：如果 warp 可以特化，为什么 GPU 不能？从 warp specialization 延伸到 block specialization，再到 GPU specialization。在 GLM-5.1 的 attention 层里，GPU 0 跑 Sparse Indexer（Top-K 选择、稀疏索引构建、路由决策），GPU 1-7 跑 MLA（RMSNorm、GEMM、Flash Sparse Attention、AllReduce）。不同阶段采用不同的扩展策略——同步密集型阶段集中在少数卡上跑，天然适合张量并行的阶段分散到多卡。

既然不同的 GPU 已经在跑不同的任务，NCCL 和 runtime 在外部编排广播和规约就显得多余了。通信自然地进入了执行流水线本身，直接嵌入 tile pipeline。整个 attention 层在 Host 层面只对应一次 Engine Kernel 启动。

性能数字：GLM-5 FP8 达到 500 tok/s，DeepSeek-V3.2 在 8×B200 上 600 tok/s（BS=1）。MTP=3 时合成负载 590 tok/s，真实生成 440 tok/s。对比 vLLM 和 SGLang 的公开基准，领先幅度不是 10% 级别的。

但我必须说几点保留。目前只支持 8×B200 硬件配置，这个限制本身不算致命——致命的是你实际要面对的东西：长短 context 交织、KV cache 碎片化、MTP accept/reject 动态改变执行流。这些在 benchmark 里看不见，在 production 里会要你的命。TileRT 团队自己也承认，从「跑得快」到「持续跑得快」，中间经历了多次执行模型重构，很多改动不增加理论吞吐，但显著改善尾部延迟。

更让我担心的是 persistent kernel 方案的灵活性。AOT 静态展开意味着你对模型结构的适配是编译期的，运行时的动态性空间有限。想想看——模型架构本身在快速演化（MoE、MTP、混合稀疏注意力），每一次架构变动都可能需要重新编排整条流水线。这不是纸面上的顾虑，是每个上生产环境的团队都会撞到的墙。

再往远看，这个方向的终局不是 TileRT 本身。TileRT 团队自己在 blog 里也写得很清楚：下一步是模型-编译器-硬件的 Co-Design。当性能逼近硬件极限，瓶颈不再是某个 GEMM 算子的优化，而是执行流水线本身的结构。模型架构、内存层次、通信拓扑、路由模式之间的冲突，无法纯靠运行时解决。这意味着推理框架需要向上渗透到模型设计阶段，向下触达硬件特性。

这个趋势我完全认同。过去十年 GPU 的角色是最大化并行计算吞吐，接下来的几年可能需要不同的东西：在极紧的延迟预算下维持执行流水线。推理系统正从「算子优化的集合」演变为「真正的 AI 执行基础设施」。

TileRT 是这个方向上目前最有说服力的尝试。但它更像是给出了正确的问题——剩下的答案，大概得靠整个生态来填。
